## Frontend Architecture

본 프로젝트의 프론트엔드는 **Next.js App Router 기반 구조**로 설계되었습니다.  
`app` 디렉토리를 중심으로 페이지를 구성하고, `components/ui`, `hooks`, `lib`, `constants`, `utils`를 분리하여 화면 로직과 공통 UI를 모듈화했습니다.

또한 `proxy.ts`를 통해 **쿠키 기반 라우트 보호**를 적용하여,
로그인 여부와 권한에 따라 접근 가능한 페이지를 분기하도록 구성했습니다.

### Structure
- **app** : 라우팅 및 페이지 구성
- **components/ui** : 공통 UI 컴포넌트
- **hooks** : 커스텀 훅
- **lib** : 공통 유틸 및 API 연동 로직
- **constants** : 상수 관리
- **utils** : 기능별 유틸 함수
- **public** : 정적 파일 관리

## Frontend Features

- **App Router 기반 페이지 구조**
- **쿠키 기반 인증 상태 처리**
- **권한별 접근 제어**
- **공통 UI 컴포넌트 분리**
- **재사용 가능한 훅 / 유틸 구조**
- **Biome + Husky 기반 코드 품질 관리**

## Frontend Routing Guard

`proxy.ts`를 기준으로 다음과 같은 접근 제어를 적용했습니다.

- **protected routes**
  - `/home`
  - `/activity`
  - `/mypage`

- **public only routes**
  - `/onboarding/auth/login`
  - `/onboarding/auth/signup`

- **admin routes**
  - `/admin`

로그인하지 않은 사용자는 보호된 페이지 접근 시 로그인 페이지로 이동하며,  
이미 로그인한 사용자는 로그인/회원가입 페이지 접근 시 홈으로 리다이렉트됩니다.  
또한 관리자 페이지는 `role` 쿠키를 기준으로 접근을 제한합니다.

## Backend Architecture

본 프로젝트의 백엔드는 **API Gateway 중심의 MSA 구조**로 설계되었습니다.  
각 도메인을 독립 서비스로 분리하고, 서비스별 DB를 분리하여 결합도를 낮췄습니다.  
서비스 간 통신은 **Feign 기반 동기 호출**과 **Kafka 기반 비동기 이벤트 처리**를 함께 사용합니다.

### Service
- **api-gateway** : 클라이언트 단일 진입점, 라우팅, 인증/인가
- **user-service** : 회원, 로그인, 사용자 기능
- **org-service** : 교회/조직 관련 기능
- **offering-service** : 헌금 및 금융 기능
- **activity-service** : 소모임 및 활동 기능
- **prayer-service** : 기도 및 신앙 활동 기능
- **common** : 공통 응답, 예외 처리, 보안, 내부 호출 공통 모듈

## Why MSA?

- 도메인별 서비스 분리로 책임 명확화
- 서비스별 DB 분리로 독립성 확보
- Gateway 기반 인증/인가 및 라우팅 일원화
- Kafka / Feign 기반 확장성과 유지보수성 향상
